# ADR-004: Bitwarden as the secrets manager via the free `bw` CLI

## Status

Accepted

## Date

2026-05-19

## Context

Both consolidated domains depend on secrets. AI tool configs hold API keys and tokens (GitHub,
the Gemini API, Context7) — some in plaintext, such as the Context7 key in the MCP master
config. The `home-lab` Ansible playbooks need credentials and passwords at run time. Once
everything is versioned in one repository, a secrets-management approach is required so that no
secret is stored in plaintext or committed.

The developer already uses Bitwarden as a personal password vault, with the Windows desktop
app, the Android app, and the command-line client. Bitwarden offers a free command-line client
(`bw`) that works with an existing personal account.

## Decision

Adopt Bitwarden as the single secrets store for all of `lm-commons`. Secrets are kept as items
in the developer's existing Bitwarden vault and accessed programmatically through the free
Bitwarden CLI (`bw`). The repository, tool configs, and Ansible playbooks reference secrets and
resolve them from the vault at run time; no secret is stored in plaintext in the repository, in
configs, or on disk.

## Alternatives Considered

### Alternative 1: SOPS + age (encrypted secrets in the repository)

- **Description**: Store secrets encrypted in-repo, decrypted on demand with an `age` key.
- **Pros**: No external service; free; secrets versioned alongside the code.
- **Cons**: Introduces a new tool and workflow the developer does not already use; the `age`
  private key still needs safekeeping outside the repository.
- **Why rejected**: The developer already relies on Bitwarden daily; reusing it avoids a new
  tool while still removing plaintext secrets.

### Alternative 2: Self-hosted vault on the homelab (Vaultwarden / HashiCorp Vault)

- **Description**: Run a secrets server on the existing Proxmox infrastructure.
- **Pros**: No SaaS cost; uses infrastructure already operated.
- **Cons**: Adds another service to maintain (uptime, backups) and a bootstrap dependency — the
  vault must be reachable to deploy anything.
- **Why rejected**: Operational overhead is not justified for a solo developer when a managed
  vault is already in use.

### Alternative 3: Bitwarden Secrets Manager (`bws`)

- **Description**: Bitwarden's separate machine-secrets product with the `bws` CLI and service
  accounts.
- **Pros**: Purpose-built for developer/machine secrets.
- **Cons**: A separate product from the Password Manager the developer already uses, with its
  own plan considerations.
- **Why rejected**: The free `bw` CLI on the existing Password Manager account already covers
  the need without adopting a separate product.

## Consequences

### Positive

- One vault holds every secret, already part of the developer's daily routine.
- No new cost and no new service to operate; desktop and mobile access included.
- Plaintext secrets are eliminated from the repository and configs.
- Rotation happens in one place.

### Negative

- A runtime dependency on the Bitwarden service and on unlocking the vault.
- Secrets are not versioned alongside the repository.

### Risks

- Unattended automation (Ansible playbooks) needs vault access without interactive unlock.
  Mitigation: use a dedicated, least-privilege CLI session for automation and confirm vault
  availability before deploys.
- Loss of the master credential locks access to all secrets. Mitigation: rely on Bitwarden's
  existing account recovery; the desktop and mobile apps provide fallback access.

## Implementation Notes

The `bw` CLI retrieves items via `bw get`; Ansible can resolve them through a Bitwarden lookup
during playbook execution. The exact retrieval and injection mechanics belong in the TechSpec.

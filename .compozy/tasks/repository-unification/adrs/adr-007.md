# ADR-007: Layered secrets — Bitwarden over a retained Ansible Vault

## Status

Accepted

## Date

2026-05-19

## Context

ADR-004 selects Bitwarden as the secrets manager. Codebase exploration found that `home-lab`
already uses Ansible Vault: `inventory/group_vars/all/vault.yml` is AES256-encrypted, and a
9-byte `.vault_pass` file (gitignored) holds the decryption password. The two mechanisms must
be combined coherently.

## Decision

Keep Ansible Vault for at-rest encryption of `group_vars` and store the Ansible Vault password
in Bitwarden. Bitwarden also holds the loose API keys and tokens (Context7, GitHub, Gemini).
The on-disk `.vault_pass` file is replaced by an executable vault-password client that
retrieves the password from Bitwarden via the `bw` CLI; `ansible.cfg` points its
`vault_password_file` directive at that client. Bitwarden is the single source of secrets;
Ansible Vault remains the encrypted-at-rest store for structured group variables.

## Alternatives Considered

### Alternative 1: Replace Ansible Vault with direct `bw` lookups

- **Description**: Remove `vault.yml`; resolve every homelab secret through a Bitwarden lookup
  in the playbooks at run time.
- **Pros**: One secrets mechanism instead of two.
- **Cons**: Discards a working, encrypted store; requires rewriting how every playbook consumes
  `group_vars`; increases per-run Bitwarden calls and disruption.
- **Why rejected**: The complementary layering removes plaintext secrets with far less change.

## Consequences

### Positive

- The on-disk `.vault_pass` file disappears; the vault password lives only in Bitwarden.
- Existing encrypted `group_vars` and playbook structure are untouched.
- One source of truth for secrets, with encryption-at-rest preserved.

### Negative

- Two mechanisms remain in play (Bitwarden and Ansible Vault), with Bitwarden bootstrapping the
  other.

### Risks

- Unattended playbook runs need the Bitwarden vault unlocked. Mitigation: the vault-password
  client uses a dedicated, least-privilege `bw` session; vault availability is checked before
  deploys.

## Implementation Notes

Ansible supports an executable `vault_password_file`: the script prints the password to stdout.
The client wraps the shared secrets-retrieval helper (`secrets.sh`).

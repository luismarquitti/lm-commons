# ADR-006: Sibling Directory Worktree Convention

## Status

Accepted

## Date

2026-05-19

## Context

Parallel work by the developer and agents requires isolated git environments to avoid conflicts and messy stashes. Git worktrees are the chosen mechanism.

## Decision

Isolated git worktrees will be created as sibling directories to the main repository root.
The naming convention will be: `../worktrees/<repo-name>-<issue-id>-<slug>`
Example: `../worktrees/intellifinance-CLE-123-add-pix-support`

## Alternatives Considered

### Alternative 1: Subdirectory (git ignored)
- **Description**: `.worktrees/<issue-id>-<slug>` inside the repo.
- **Pros**: Keeps everything in one parent folder.
- **Cons**: Some tools or IDEs might still index them; risk of accidental inclusion in searches.
- **Why rejected**: User preferred sibling directories for cleaner separation.

### Alternative 2: Managed by a central tool
- **Description**: A tool that handles a configurable base path.
- **Pros**: More flexible.
- **Cons**: Adds another layer of configuration.
- **Why rejected**: Sibling directories are simple and effective for now.

## Consequences

### Positive
- Clear separation between the main branch and feature work.
- Easy to navigate between multiple active tasks.
- No risk of accidental commits of worktree files to the main branch.

### Negative
- Requires the user to navigate "up and down" directories.

### Risks
- Path length limits on some systems (though unlikely to be an issue on Linux).

## Implementation Notes

The workflow agent should ensure `../worktrees/` directory exists before creation.

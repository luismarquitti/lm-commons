# ADR-010: Agent-Managed Worktree Lifecycle

## Status

Accepted

## Date

2026-05-19

## Context

To enable parallel work, each task should run in its own git worktree. We need to decide who manages these worktrees.

## Decision

The Developer Agent assigned to a task is responsible for the full lifecycle of its worktree:
1. Creating the worktree based on the Linear Issue ID and title.
2. Performing the work.
3. Deleting the worktree once the Pull Request has been merged into the main branch.

## Alternatives Considered

### Alternative 1: Developer-managed
- **Description**: The human creates the worktree and hands it to the agent.
- **Pros**: Developer has full control over the local filesystem.
- **Cons**: High manual overhead for the developer.
- **Why rejected**: User wants the agent to handle the isolation logic.

## Consequences

### Positive
- Fully automated workspace isolation for agents.
- Reduced manual steps for the orchestrator.
- Ensures a clean environment for every task.

### Negative
- Agents need permissions and logic to run `git worktree` commands safely.

### Risks
- Failed deletions might lead to "ghost" worktrees taking up space or causing conflicts.

## Implementation Notes

The `cy-execute-task` or a new orchestration skill should include steps for worktree management.

# ADR-011: Pilot the workflow on a new greenfield IntelliFinance feature

## Status

Accepted

## Date

2026-05-20

## Context

All Phase 1 infrastructure tasks (01–06) are complete: the PLAYBOOK is documented, AGENTS.md
is in place, `sync-compozy-tasks.py` (with tests) synchronizes Compozy specs to Linear, the
`worktree-manager.sh` script manages isolated parallel workspaces, and the Linear skill has been
updated to English and extended for SDD integration.

The sole remaining MVP gate is an end-to-end run of the full workflow. The `financial-insights-
dashboard` feature in IntelliFinance was implemented via the old conductor workflow before the
new SDD pipeline existed, so it does not validate the new process. A pilot is needed to prove
the pipeline works end-to-end with a real feature.

Two options were considered for the pilot: retrofit the existing `financial-insights-dashboard`
tasks through the new pipeline, or pick a new small greenfield feature. The greenfield approach
was selected.

## Decision

Run the end-to-end pilot on a new, scoped IntelliFinance feature — chosen at the time of
execution — that has not been started before. The feature must flow through every step of the
PLAYBOOK: idea → `cy-create-prd` → `cy-create-techspec` → `cy-create-tasks` →
`sync-compozy-tasks.py` → Linear → worktree → implementation → PR → `/cy-review-round` →
human merge. The developer agent runs locally via Claude Code inside the isolated worktree.

## Alternatives Considered

### Alternative 1: Retrofit financial-insights-dashboard

- **Description**: Sync the existing `financial-insights-dashboard` task files to Linear, open
  PRs for remaining tasks, and run the review gate on them.
- **Pros**: No new spec work needed; tasks are already defined.
- **Cons**: Does not validate the spec pipeline (PRD/TechSpec/tasks were created outside the
  new SDD flow); the existing tasks were done with conductor, so the trail is mixed; the start
  of the pipeline is never exercised.
- **Why rejected**: A retrofit validates only part of the pipeline. The spec creation steps —
  the most important part to validate — would be skipped.

### Alternative 2: Bug fix or small enhancement

- **Description**: Pick an existing open bug or small enhancement as the pilot.
- **Pros**: Lower spec overhead; faster to complete.
- **Cons**: Bugs rarely justify a full PRD/TechSpec cycle; validating the spec pipeline on a
  bug is not representative of normal feature work.
- **Why rejected**: The spec pipeline is the core of the workflow; the pilot should exercise it
  fully.

## Consequences

### Positive

- The entire pipeline — spec, Linear sync, worktree, PR, review gate, merge — is validated on
  a real feature.
- The greenfield feature produces durable spec artifacts that follow the standard template.
- A successful pilot proves the workflow is repeatable without ad-hoc deviation.

### Negative

- Requires selecting and speccing a new IntelliFinance feature before the pilot can begin.
- The pilot takes slightly longer than a retrofit would.

### Risks

- The pilot feature may reveal gaps in the PLAYBOOK or tooling that require iteration before
  the pilot can complete. Mitigation: keep the pilot feature small and well-scoped.

## Implementation Notes

The pilot feature should be selected at the start of task_07 execution. It must be small enough
to complete in a single iteration of the spec pipeline. The developer agent runs locally in a
worktree created by `worktree-manager.sh`. The reviewer agent runs via `/cy-review-round`.

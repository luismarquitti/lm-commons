# PRD: Mission Control (Home Lab Web Portal)

## Overview
**Mission Control** is a centralized web portal designed to standardize the creation, hosting, and viewing of dashboards, reports, and investigations within the Home Lab ecosystem. It serves as a polished interface for both fixed system monitoring (Linear tasks, Agent status) and dynamic, agent-generated content (daily plans, research reports). By providing an API-driven "Agentic Shell," it enables OpenClaw agents to deploy professional-grade web pages with minimal effort while maintaining a consistent design language.

## Goals
*   **Standardization:** Establish a single React + TypeScript base for all internal web interfaces.
*   **Agent Empowerment:** Provide a simple JSON API for agents to generate polished pages using pre-defined templates.
*   **Accessibility:** Ensure all dashboards are accessible across the home lab network via Tailscale.
*   **Knowledge Integration:** Enable seamless distillation of temporary web content into the OpenClaw memory system.

## User Stories
### Persona: Developer (Luis)
*   **As a Developer,** I want a unified "Mission Control" dashboard so that I can see my Linear tasks and the status of my home lab agents in one place.
*   **As a Developer,** I want to use my Tailscale network names to access my tools so that I don't have to manage complex public DNS or SSL for internal-only apps.
*   **As a Developer,** I want to test my applications (IntelliFinance, ClinicCare) in a standardized environment within my network.

### Persona: OpenClaw Agent
*   **As an Agent,** I want to "publish" my daily plan as a polished web page so that my human collaborator can view it easily.
*   **As an Agent,** I want to deploy a custom research investigation as a self-contained bundle when the standard templates aren't enough.

## Core Features
### 1. The Agentic Shell (React Hub)
*   The main web application built with React, TypeScript, and Vanilla CSS.
*   **Navigation:** A sidebar/menu to switch between fixed dashboards and a list of active dynamic pages.

### 2. Template Registry & API
*   **API Endpoint:** A REST/Webhook listener that accepts JSON payloads (e.g., `{ template: "Investigation", data: { ... } }`).
*   **Standard Templates:** Initial set including `DailyPlan`, `LinearTaskBoard`, and `AgentStatus`.

### 3. Static Hosting "Escape Hatch"
*   Ability to host and serve raw HTML/React bundles uploaded by agents to a specific path (e.g., `/portal/custom/investigation-xyz/`).

### 4. Dynamic Page Management
*   **Temporary vs. Fixed:** Support for ephemeral pages that are meant to be distilled and eventually archived.
*   **Metadata:** Each dynamic page includes metadata (creation date, author agent, tags) for easy searching.

## User Experience
*   **Primary Flow:** Agent performs a task -> Agent POSTs results to Portal -> Portal generates a unique URL -> Agent provides URL to Luis.
*   **UI/UX:** Focus on a clean, "Mission Control" aesthetic—high information density but organized. Interactive elements (collapsible sections, filterable tables) are standard in templates.

## High-Level Technical Constraints
*   **Tech Stack:** React 19+, TypeScript, Vanilla CSS.
*   **Networking:** Accessible via Tailscale (`http://[NODE_NAME]:[PORT]`). No public domain/SSL required for MVP.
*   **Integration:** Must expose an endpoint compatible with OpenClaw agent tool calls.

## Non-Goals
*   **Public Internet Access:** The portal is strictly for internal home lab use.
*   **Complex Auth:** Initially relying on Tailscale's network security; no per-user login/passwords in Phase 1.
*   **Memory Distillation Logic:** The Portal *provides* the data; the *logic* for distillation remains inside OpenClaw's heartbeat.

## Phased Rollout Plan
### Phase 1: MVP (The Foundation)
*   Scaffold React Shell with Vanilla CSS.
*   Implement `LinearTaskBoard` and `AgentStatus` fixed dashboards.
*   API endpoint for the `DailyPlan` template.
*   **Success Criteria:** Luis can view his "Plan of the Day" generated by an agent via a Tailscale URL.

### Phase 2: Custom Bundles & History
*   Implement the static hosting "Escape Hatch."
*   Create a "History/Archive" view for old reports.
*   **Success Criteria:** Agents can deploy a custom Three.js or complex React bundle that the Portal serves.

### Phase 3: Memory Integration
*   Extend OpenClaw heartbeat to scan the Portal's dynamic pages.
*   Implement "Pinned" vs. "Disposable" page logic.
*   **Success Criteria:** Temporary pages are automatically distilled into `MEMORY.md` before cleanup.

## Success Metrics
*   **Usage:** 100% of OpenClaw "investigations" are published to the Portal.
*   **Performance:** Portal loads in <500ms on the local network.
*   **Reliability:** 0% failure rate for JSON-to-Template rendering.

## Risks and Mitigations
*   **Risk:** Portal becomes a "dumping ground" for useless agent pages. **Mitigation:** Strict TTL (Time-To-Live) and distillation logic via OpenClaw heartbeat.
*   **Risk:** Template development slows down feature delivery. **Mitigation:** Prioritize the "Escape Hatch" so agents can always fall back to raw HTML.

# ADR-002: Node.js + Express as the API Backend

## Status

Accepted

## Date

2026-05-20

## Context

Mission Control requires a backend process to receive JSON payloads from OpenClaw agents (template rendering requests), persist dynamic page metadata, and serve static file bundles for the escape hatch. The frontend is React + TypeScript built with Vite. A backend technology must be chosen that minimises operational complexity for a single-developer home lab while fitting the existing Node.js/npm toolchain already used by OpenClaw on lm-claw.

## Decision

Use Node.js with Express as the API backend. The backend and frontend share a single `package.json`, TypeScript configuration, and deployment process. Express handles all `/api/*` routes; the Vite-built React output is served as static files by Express in production.

## Alternatives Considered

### Alternative 1: Go HTTP Server

- **Description**: Compiled Go binary serving the API; React is pre-built and served as separate static assets.
- **Pros**: Strong type safety; lower memory footprint; single compiled binary.
- **Cons**: Introduces a second language to an otherwise JavaScript/TypeScript stack; increases cognitive overhead; requires separate build steps.
- **Why rejected**: No existing Go code in the repository. Adding Go solely for a home lab portal is not justified by the scale or complexity of the API surface.

### Alternative 2: Hono (Node.js, TypeScript-native)

- **Description**: Modern lightweight TypeScript HTTP framework running on Node.js or Bun.
- **Pros**: Smaller bundle than Express; better TypeScript ergonomics; edge-runtime compatible.
- **Cons**: Less documentation and community tooling than Express; fewer middleware options; unfamiliar territory without prior project precedent.
- **Why rejected**: Express ecosystem maturity and tutorial availability outweigh Hono's marginal size advantage for an internal home lab portal.

## Consequences

### Positive

- Single language (TypeScript) across the full stack.
- One `npm install`, one `npm run dev`, one systemd unit.
- Large ecosystem of middleware (multer for file uploads, better-sqlite3, etc.).

### Negative

- Node.js process required at runtime (no static-only deployment).
- Express is not strictly typed by default; requires discipline or additional validation middleware (e.g., zod).

### Risks

- **Single process failure**: If Express crashes, both the API and static serving go down. Mitigation: systemd `Restart=on-failure` ensures automatic recovery.

## Implementation Notes

- Use `tsx` for TypeScript-native execution in development (`tsx watch server/index.ts`).
- Compile to `dist/server/` with `tsc` for production.
- Validate incoming agent payloads with `zod` schemas to avoid runtime surprises.

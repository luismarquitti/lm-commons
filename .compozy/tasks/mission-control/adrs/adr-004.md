# ADR-004: Single-Port Vite + Express Serving Model

## Status

Accepted

## Date

2026-05-20

## Context

Mission Control runs a Vite-based React frontend and an Express API backend. These two development-time servers must coexist without CORS complications, and the production deployment must be simple enough to manage with a single systemd unit on lm-claw. A serving model must be chosen that works in both development and production without requiring Nginx or a reverse proxy in front of the application.

## Decision

Use Vite's built-in proxy configuration in development so that all `/api/*` requests are forwarded from the Vite dev server (port 5173) to the Express process (port 3001). In production, Express serves the Vite-built static output from `dist/client/` via `express.static`, so all traffic hits a single port (e.g., 3001 or configurable via `PORT` env var). One process, one port, one systemd unit.

## Alternatives Considered

### Alternative 1: Separate Ports — Vite on :5173, Express on :3001

- **Description**: Always run two separate processes; the React app makes cross-origin requests to Express.
- **Pros**: Conceptual simplicity — each server has a single responsibility.
- **Cons**: Requires CORS headers on all Express routes; requires two systemd units in production; agents must know two different URLs.
- **Why rejected**: CORS configuration is error-prone and unnecessary when a proxy already handles routing at the serving layer.

### Alternative 2: Nginx Proxy Manager as Reverse Proxy

- **Description**: Route `/api` to Express and `/` to the Vite static build via the existing Nginx Proxy Manager LXC (192.168.3.7).
- **Pros**: Clean separation; Nginx handles static file caching and compression.
- **Cons**: Adds infrastructure dependency; requires configuration changes in Nginx Proxy Manager whenever port or path changes; complicates local development.
- **Why rejected**: Introduces an external dependency for what is a straightforward single-application serving scenario. The home lab already has Uptime Kuma for monitoring; adding Nginx routing for a single internal app is over-engineering.

## Consequences

### Positive

- Single URL for both the portal UI and the agent API (`http://lm-claw:3001`).
- No CORS configuration required.
- One systemd unit (`mission-control.service`) manages the entire application.
- Development workflow mirrors production routing behaviour.

### Negative

- Express serves static files in production, which is less performant than Nginx for high-throughput static content. Acceptable at home lab scale (<10 concurrent users).
- Vite proxy config must be kept in sync with Express port if the default changes.

### Risks

- **Port conflict**: If port 3001 is already in use on lm-claw, the server fails silently. Mitigation: configure port via `PORT` environment variable; document default in README.

## Implementation Notes

`vite.config.ts` proxy configuration:
```ts
server: {
  proxy: {
    '/api': 'http://localhost:3001',
    '/portal': 'http://localhost:3001',
  }
}
```

Production Express static serving:
```ts
app.use(express.static(path.join(__dirname, '../client')));
app.get('*', (_, res) => res.sendFile(path.join(__dirname, '../client/index.html')));
```

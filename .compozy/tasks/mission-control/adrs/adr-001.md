# ADR-001: Adopt the Agentic Shell Approach for Home Lab Web Portal

## Status

Accepted

## Date

2026-05-20

## Context

The user needs a standardized way to create and view web-based reports, investigations, and dashboards within the OpenClaw and Home Lab ecosystem. The solution must support fixed dashboards (e.g., Linear tasks, Agent status) and dynamic, agent-generated content (e.g., "Plan of the Day"). These pages must be accessible across the home lab network via Tailscale.

## Decision

We will implement the "Agentic Shell" approach. This involves a central React + TypeScript web application that serves as the "Web Portal." 

Key architectural components:
- **Central Service**: A unified web portal running on the home lab infrastructure.
- **API-First Rendering**: The portal exposes a JSON API for agents to send data to pre-defined React templates.
- **Custom Bundles**: The portal supports hosting self-contained HTML/React bundles for custom agent-generated UI that doesn't fit standard templates.
- **Network Access**: Services will be accessed via direct IP or Tailscale names (e.g., `http://optiplex:PORT`) to maintain simplicity.
- **Memory Integration**: OpenClaw's existing memory heartbeat will be extended to monitor the Portal's dynamic pages for distillation, rather than the Portal managing its own memory loop.

## Alternatives Considered

### Alternative 1: Infrastructure-as-Service (Decentralized)

- **Description**: A lightweight static host where each tool/agent is responsible for its own full rendering (HTML/JS/CSS).
- **Pros**: Maximum freedom for agents; decoupled architecture.
- **Cons**: High visual fragmentation; increased complexity for agents to handle rendering logic; harder to maintain a consistent "Mission Control" experience.
- **Why rejected**: The user preferred a standardized "look and feel" and an API-driven model for ease of use by agents.

### Alternative 2: Public Domain + Reverse Proxy

- **Description**: Purchasing a public domain and setting up a full DNS/Reverse Proxy infrastructure.
- **Pros**: Professional URLs; easier SSL management; accessibility without Tailscale (if desired).
- **Cons**: Higher setup complexity and cost; potential security surface area increase.
- **Why rejected**: The user explicitly chose to stay with Direct IP/Tailscale names to prioritize simplicity in networking for now.

## Consequences

### Positive

- Standardized UI/UX across all home lab dashboards.
- Lower barrier for agents to generate polished reports (simple JSON payload).
- Centralized hosting for all internal tools.
- Clean separation of concerns between content creation (Agents) and rendering/hosting (Portal).

### Negative

- Requires maintaining a "Template Registry" within the Portal application.
- Initial setup overhead for the React Shell.

### Risks

- **Template Rigidity**: Agents might find JSON templates too restrictive. Mitigation: Allow "Escape Hatch" for custom HTML/React bundles.
- **Single Point of Failure**: If the Portal service goes down, all dashboards are inaccessible. Mitigation: Standard Proxmox/Ansible monitoring and backup procedures.

## Implementation Notes

- Use React + TypeScript + Vanilla CSS (per user preference).
- Implement a simple "Host" endpoint that accepts file uploads for custom bundles.
- Ensure the API is lightweight and easy for OpenClaw agents to invoke.

# ADR-003: SQLite for Dynamic Page Metadata Storage

## Status

Accepted

## Date

2026-05-20

## Context

Dynamic pages created by agents require persistent metadata: title, template type, creation timestamp, author agent identifier, TTL, tags, and the JSON payload used to render the page. This metadata must survive process restarts, support basic queries (filter by tag, list by date, check TTL expiry), and be available without any external service dependency.

## Decision

Use SQLite via the `better-sqlite3` npm package. A single `.db` file lives on the lm-claw filesystem (e.g., `/var/lib/mission-control/data.db`). All reads and writes are synchronous, which fits Express's request/response model without requiring async database drivers.

## Alternatives Considered

### Alternative 1: PostgreSQL (192.168.3.20)

- **Description**: Use the existing PostgreSQL LXC container on pve-inspiron.
- **Pros**: Full SQL feature set; central location reachable by other services; familiar tooling.
- **Cons**: Network dependency — if the pve-inspiron node is down or unreachable, Mission Control cannot read or write page metadata. Adds latency to every request. Requires connection pooling (pg-pool) and async query handling.
- **Why rejected**: A home lab web portal should not fail to serve existing pages because a remote database LXC is unreachable. SQLite eliminates the network hop entirely.

### Alternative 2: JSON Flat Files

- **Description**: One JSON file per page stored in a directory (e.g., `/var/lib/mission-control/pages/`).
- **Pros**: No driver required; human-readable; trivial to back up.
- **Cons**: Filtering and sorting require reading all files into memory; no atomic transactions; file-system race conditions under concurrent writes.
- **Why rejected**: Query operations (list by date, filter by tag, TTL expiry scan) become error-prone and slow as page count grows.

### Alternative 3: In-Memory Only

- **Description**: Pages live in Node.js process memory; lost on restart.
- **Pros**: Zero setup; maximum speed.
- **Cons**: Agents must re-generate all pages after every server restart, which is unacceptable for daily plans and investigation reports.
- **Why rejected**: Data durability is a hard requirement.

## Consequences

### Positive

- Zero external dependencies: Mission Control is fully self-contained on lm-claw.
- Synchronous `better-sqlite3` API simplifies Express route handlers (no async/await for DB calls).
- SQLite file is trivially backed up with `cp` or included in Ansible snapshot procedures.

### Negative

- Not suitable for concurrent writes from multiple processes; single-writer constraint.
- Migration tooling (schema changes) must be managed manually or via a lightweight migration library.

### Risks

- **File corruption on crash**: Mitigated by SQLite's WAL (Write-Ahead Logging) mode, which is enabled by default in `better-sqlite3`.
- **Schema evolution**: As templates grow, the pages table may need new columns. Mitigation: store the full JSON payload as a TEXT column alongside structured metadata fields, minimising future schema changes.

## Implementation Notes

- Enable WAL mode on database open: `db.pragma('journal_mode = WAL')`.
- Store the raw JSON payload in a `payload TEXT` column; structured fields (template, author, ttl) as typed columns.
- Run schema initialisation via an `initDb()` function called at server startup.
